## 2. Введение

### 2.1. Цель документа

Данный тест-план описывает подход, объём, ресурсы и расписание тестирования программной системы «СУППЗ» (Система учёта почтовых пересылок из-за границы). Документ разработан согласно стандарту IEEE 829-1998.

### 2.2. Предмет тестирования

Тестируется система СУППЗ, реализующая:
- Регистрацию входящих международных пересылок (входная форма)
- Расчёт и учёт таможенных пошлин
- Управление статусами пересылок
- Фиксацию оплаты и выдачи
- Формирование извещений получателям (выходная форма)
- Учёт невостребованных отправлений
- Аутентификацию и разграничение прав доступа

## 6. Подход к тестированию (Approach)

### 6.1. Виды тестирования

| Вид тестирования | Описание | Инструменты |
|-----------------|----------|------------|
| **Функциональное тестирование** | Проверка функций на соответствие SRS | Ручное тестирование, Playwright |
| **Тестирование граничных значений** | Поля форм: трек-номер, стоимость (ровно на пороге), вес (максимум 31.5 кг) | Ручное |
| **Негативное тестирование** | Некорректные данные, пустые поля, неверный трек-номер | Ручное |
| **Тестирование переходов состояний** | Все допустимые и недопустимые переходы статусов пересылки | Ручное |
| **Тестирование прав доступа** | Ролевая модель (Оператор / Старший / Администратор) | Ручное |
| **Нагрузочное тестирование** | 10 пользователей, время отклика ≤ 3 сек (PR-01) | Apache JMeter |
| **Тестирование PDF-вывода** | Корректность содержимого извещения (с пошлиной и без) | Ручное |
| **Кроссбраузерное тестирование** | Chrome, Firefox, Edge | Ручное / Playwright |

### 6.2. Техника тест-дизайна

- **Классы эквивалентности:** для трек-номера (корректный S10, некорректный, пустой); стоимости (ниже порога, ровно на пороге, выше порога); статусов.
- **Граничные значения:** стоимость ровно на пороге беспошлинного ввоза; вес = 31.5 кг (максимум ВПС); срок хранения: за 5 дней до и в день истечения.
- **Диаграмма переходов состояний:** для тестирования всех допустимых/недопустимых переходов статусов пересылки.
- **Сценарное тестирование (E2E):** полный жизненный цикл пересылки — от регистрации до выдачи; цикл с пошлиной и без пошлины.

### 6.3. Приоритеты тестирования

1. Полный E2E цикл: регистрация → расчёт пошлины → оплата → выдача
2. Расчёт пошлины на граничных значениях
3. Переходы статусов (допустимые и запрещённые)
4. Формирование PDF-извещения
5. Разграничение прав доступа

## 7. Критерии прохождения и провала тестов

### 7.1. Уровень отдельного тест-кейса

**ПРОЙДЕН:** фактический результат совпадает с ожидаемым.

**ПРОВАЛЕН:** фактический результат не совпадает с ожидаемым, или система завершилась с ошибкой / зависанием.

### 7.2. Критерии завершения тестирования

| Уровень | Критерий прохода | Критерий провала |
|---------|-----------------|-----------------|
| **Критический** | 100% тест-кейсов приоритета «Высокий» — пройдено | Любой открытый blocker-дефект |
| **Средний** | ≥ 95% всех тест-кейсов пройдено | Более 3 дефектов уровня «Высокий» открыто |
| **Приёмочный** | ≤ 2% тест-кейсов провалено | Любой заблокированный дефект |
