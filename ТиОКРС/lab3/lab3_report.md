## Часть 1. Анализ предметной области

### 1.1. Задачи по обработке информации в предметной области

Предметная область — **учёт почтовых пересылок из-за границы** — охватывает деятельность почтовых отделений, таможенных органов и логистических служб, принимающих международные почтовые отправления (посылки, письма, бандероли). Основные задачи обработки информации:

1. **Регистрация входящей международной пересылки.** При поступлении международного отправления оператор фиксирует: трек-номер, страну и город отправления, данные отправителя и получателя, вид отправления, объявленную стоимость, вес, дату поступления. Присваивается внутренний идентификатор.

2. **Таможенное оформление.** Для посылок стоимостью выше порога беспошлинного ввоза рассчитывается и фиксируется размер таможенной пошлины. Оператор регистрирует факт таможенного досмотра, дату и его результат.

3. **Учёт оплаты пошлин и сборов.** Фиксируется факт оплаты получателем таможенной пошлины и/или хранения: дата оплаты, сумма, способ оплаты. После оплаты отправление переходит в статус «Готово к выдаче».

4. **Уведомление получателя.** Формирование и отправка извещения получателю (бумажное в почтовый ящик или СМС/e-mail) с указанием трек-номера, суммы к оплате (при наличии) и срока хранения.

5. **Выдача отправления.** Получатель предъявляет извещение и документ, удостоверяющий личность. Оператор фиксирует факт выдачи: дата, данные получателя, статус меняется на «Выдано».

6. **Учёт отправлений с истёкшим сроком хранения.** Отправления, не востребованные в течение установленного срока (как правило, 30 суток), переводятся в статус «Возврат» и отправляются обратно отправителю.

7. **Формирование отчётности.** Сводные отчёты за период: количество поступивших пересылок по странам отправления, суммы собранных пошлин, количество невостребованных отправлений.

### 1.2. Существующие стандарты для документов и форм

В сфере международных почтовых пересылок применяются следующие нормативные документы и стандарты:

| Документ | Описание |
|----------|----------|
| **Акты Всемирного почтового союза (ВПС / UPU)** | Регламентируют порядок обработки международных почтовых отправлений, форматы данных, трек-номера (стандарт S10) |
| **Таможенная декларация CN22** | Упрощённая декларация для мелких пакетов стоимостью до 300 SDR |
| **Таможенная декларация CN23** | Полная декларация для посылок; содержит: описание содержимого, количество, вес, стоимость, HS-код |
| **Почтовое извещение форма 22** (РБ/РФ) | Бумажное уведомление получателю о поступлении отправления |
| **Таможенный кодекс ЕАЭС** | Регулирует порядок таможенного декларирования и уплаты пошлин |
| **ГОСТ Р 51207-98** | Услуги почтовой связи; общие требования |
| **S10 UPU** | Международный стандарт трек-номера (2 буквы + 8 цифр + контрольная + 2 буквы страны) |

**Форма извещения получателю включает:**
- Трек-номер отправления
- Страна отправления, наименование отправителя
- Вид отправления (посылка, мелкий пакет, письмо)
- Объявленная стоимость, вес
- Сумма к оплате (пошлина + сбор за хранение при наличии)
- Дата поступления, крайний срок хранения
- Адрес почтового отделения, режим работы

| Система | Разработчик / Оператор | Решаемые задачи |
|---------|----------------------|-----------------|
| **Белпочта (РБ)** — tracking.belpost.by | РУП «Белпочта» | Отслеживание по трек-номеру, статусы доставки, электронные уведомления |

**Выявленные недостатки** для небольших региональных почтовых отделений:
- Системы крупных операторов (Белпочта, Почта России) не предоставляют API для внутренней автоматизации конкретного отделения
- 1С:Почтовый оператор — высокая стоимость, сложная настройка
- Отсутствие удобных инструментов для расчёта и учёта таможенных платежей «на месте»

## 1. Введение

### 1.1. Назначение документа

Данный документ является спецификацией требований к программному обеспечению (SRS) для системы «СУППЗ» — системы учёта почтовых пересылок из-за границы. SRS составлен в соответствии с рекомендациями стандарта IEEE Std 830-1998.

Документ предназначен для:
- Разработчиков программного обеспечения системы
- Команды тестирования
- Заказчика — руководства почтового отделения

### 1.2. Область применения

Программный продукт «СУППЗ» предназначен для автоматизации процесса регистрации международных почтовых пересылок, поступающих в почтовое отделение, учёта таможенных платежей, управления статусами и формирования извещений получателям.

**Система позволяет:**
- Регистрировать входящие международные пересылки (посылки, мелкие пакеты, письма с объявленной ценностью)
- Рассчитывать и фиксировать таможенные пошлины и сборы
- Отслеживать статусы пересылок (поступила → на таможне → ожидает оплаты → готова к выдаче → выдана / возврат)
- Фиксировать факт оплаты пошлины получателем
- Формировать и печатать извещения получателям
- Регистрировать выдачу отправлений
- Вести реестр невостребованных пересылок
- Формировать сводные отчёты

**Система не охватывает:** международный обмен данными с зарубежными почтовыми операторами, интеграцию с государственными таможенными системами (в данной версии).

## 2. Общее описание

### 2.1. Перспектива продукта

СУППЗ является автономным веб-приложением, работающим в рамках локальной сети почтового отделения. Система взаимодействует со следующими внешними элементами:

- **Справочник адресов и индексов** (только чтение, для автозаполнения)
- **Принтер:** для вывода извещений получателям (формат A5 / A6)
- **Настройки таможенных порогов:** редактируются администратором по актуальным нормам

На данном этапе автоматический обмен статусами с внешними системами слежения не предусмотрен.

- Доступ — только через локальную сеть отделения.

- Язык интерфейса — русский.
- Поддержка офлайн-режима (локальная БД) при потере соединения с сервером не требуется.

- Принтер подключён к рабочему месту оператора.

## 3. Детальные требования

### 3.1. Требования к внешним интерфейсам

#### 3.1.1. Интерфейс пользователя

- Веб-интерфейс, оптимизированный для разрешения 1366×768+.
- Главное меню: «Пересылки», «Реестр», «Извещения», «Отчёты», «Настройки».
- Поддержка ввода трек-номера с физического сканера штрих-кодов (USB HID).
- Таблица статусов выделяется цветами: «На таможне» — жёлтый, «Ожидает оплаты» — оранжевый, «Готова к выдаче» — зелёный, «Возврат» — красный.

#### 3.1.2. Интерфейс программного обеспечения

- СУБД: PostgreSQL 14+.
- Протокол: HTTPS.
- Формат экспорта отчётов: PDF, XLSX.

#### 3.1.3. Интерфейс оборудования

- USB штрих-код сканер (стандарт HID, эмуляция клавиатуры).
- Принтер А4 для извещений.

### 3.2. Функциональные требования

#### F-01. Регистрация входящей пересылки

**FR-01.1** Система должна предоставлять форму регистрации пересылки (см. спецификацию входной формы в разделе 3.6.1).

**FR-01.2** При сохранении система присваивает уникальный внутренний номер вида `PS-YYYYMMDD-NNNNN`.

**FR-01.3** Дата и время поступления фиксируются автоматически.

**FR-01.4** Если объявленная стоимость превышает порог беспошлинного ввоза (текущее значение из настроек), система автоматически рассчитывает ориентировочный размер пошлины и устанавливает статус «На таможне».

**FR-01.5** Если объявленная стоимость ниже порога — статус сразу «Готова к выдаче», пошлина = 0.

**FR-01.6** Обязательные поля валидируются перед сохранением.

#### F-02. Поиск и просмотр пересылки

**FR-02.1** Поиск по: внутреннему номеру, трек-номеру, ФИО получателя, дате поступления (диапазон), статусу.

**FR-02.2** Результаты поиска — таблица: Внутр. №, Трек-номер, ФИО получателя, Страна, Вид, Статус, Дата поступления.

#### F-03. Расчёт и учёт таможенной пошлины

**FR-03.1** Автоматический расчёт пошлины по формуле, заданной администратором (например, 15% от суммы превышения порога).

**FR-03.2** Оператор может скорректировать рассчитанную пошлину вручную с обязательным указанием причины.

#### F-04. Фиксация оплаты пошлины

**FR-04.1** После оплаты получателем оператор фиксирует: дату оплаты, сумму, способ оплаты. Статус меняется на «Готова к выдаче».

#### F-05. Извещение получателю

**FR-05.1** Форма извещения автоматически заполняется данными из записи о пересылке.

**FR-05.2** Система поддерживает печать извещения и экспорт в PDF.

**FR-05.3** Извещение содержит все поля, описанные в спецификации выходной формы (раздел 3.6.2).

#### F-06. Регистрация выдачи

**FR-06.1** Оператор фиксирует выдачу: дата/время, серия и номер документа получателя. Статус → «Выдано».

**FR-06.2** Выдача возможна только для пересылок со статусом «Готова к выдаче».

#### F-07. Управление статусами

**FR-07.1** Допустимые переходы статусов:

```
Поступила → На таможне → Ожидает оплаты → Готова к выдаче → Выдано
Поступила → Готова к выдаче → Выдано
Любой статус (кроме «Выдано») → Возврат (по истечении срока хранения или вручную)
```

**FR-07.2** Ручная смена статуса оператором требует указания причины.

#### F-08. Учёт невостребованных пересылок

**FR-08.1** Система автоматически помечает пересылки, не выданные в течение N суток (N настраивается, по умолчанию 30), флагом «Срок истекает» за 5 суток до окончания.

**FR-08.2** По истечении срока статус меняется на «Возврат». Формируется запись о возврате.

#### F-09. Отчётность

**FR-09.1** Отчёт «Реестр поступивших пересылок за период»: дата диапазон, фильтр по стране / статусу.

**FR-09.2** Отчёт «Сводка таможенных платежей за период»: количество оплаченных пошлин, общая сумма.

#### F-10. Аутентификация и авторизация

**FR-10.1** Вход по логину и паролю.

**FR-10.2** Роли: «Оператор», «Старший оператор», «Администратор».

**FR-10.3** Оператор не имеет доступа к отчётности и настройкам пошлин.

### 3.3. Требования к производительности

**PR-01.** Время отклика при стандартных операциях — не более 3 секунд при до 10 одновременных пользователей.

**PR-02.** Формирование PDF-извещения — не более 3 секунд.

**PR-03.** База данных — не менее 500 000 записей без деградации поиска.

### 3.4. Проектные ограничения

- Open-source или лицензионно-чистое ПО.
- PostgreSQL для СУБД.
- Развёртывание на Linux (Ubuntu 22.04 LTS) или Docker.

### 3.5. Атрибуты качества

| Атрибут | Требование |
|---------|-----------|
| **Надёжность** | Доступность ≥ 99% в рабочее время |
| **Безопасность** | HTTPS, пароли — bcrypt, логирование действий |
| **Сопровождаемость** | Структурированный код, документация |

### 3.6. Спецификации форм

#### 3.6.1. Входная форма — «Регистрация входящей международной пересылки»

**Назначение:** Фиксация данных о международном почтовом отправлении при его поступлении в почтовое отделение.

**Пользователь:** Оператор почтового отделения.

| № | Поле | Тип | Обязательное | Ограничения / Правила |
|---|------|-----|:---:|----------------------|
| 1 | Внутренний номер | Строка (авто) | — | Генерируется автоматически: `PS-YYYYMMDD-NNNNN` |
| 2 | Дата и время поступления | Дата/время (авто) | — | Текущие дата/время сервера, не редактируются |
| 3 | Трек-номер | Строка | ✓ | Стандарт S10 UPU: 2 буквы + 8 цифр + контрольная + 2 буквы; поддержка сканера |
| 4 | Страна отправления | Выбор из справочника | ✓ | Список стран ISO 3166-1 |
| 5 | Наименование отправителя | Строка | ✓ | Макс. 100 символов |
| 6 | Вид отправления | Перечисление | ✓ | «Посылка» / «Мелкий пакет» / «Письмо с объявленной ценностью» / «EMS» |
| 7 | Объявленная стоимость | Число | ✓ | Дробное ≥ 0; по умолчанию 0 (нет декларированной стоимости) |
| 8 | Валюта стоимости | Перечисление | ✓ | EUR / USD / BYN; по умолчанию EUR |
| 9 | Вес (кг) | Число | ✓ | Дробное > 0; макс. 31.5 (ограничение ВПС) |
| 10 | ФИО получателя | Строка | ✓ | Макс. 150 символов |
| 11 | Адрес получателя | Строка | ✓ | Макс. 255 символов |
| 12 | Телефон получателя | Строка | — | Формат +375XXXXXXXXX; для СМС-уведомления |
| 13 | Наименование вложения (по CN22/CN23) | Строка | — | Краткое описание, макс. 200 символов |
| 14 | Примечание | Текст | — | Макс. 500 символов |

**Автоматические действия при сохранении:**
- Если поле 7 (стоимость) > порог беспошлинного ввоза → расчёт пошлины (FR-03.1), статус «На таможне».
- Если стоимость ≤ порога или = 0 → статус «Готова к выдаче».

**Действия на форме:**
- **«Сохранить»** — валидация, запись в БД.
- **«Очистить»** — сброс введённых данных.
- **«Отмена»** — закрытие без сохранения.

**Ошибки валидации:**
- Некорректный трек-номер → «Неверный формат трек-номера (стандарт S10)»
- Не указана страна → «Укажите страну отправления»
- Вес ≤ 0 → «Вес должен быть положительным числом»
- Пустой получатель → «Укажите ФИО получателя»

---

#### 3.6.2. Выходная форма — «Извещение получателю о поступлении международной пересылки»

**Назначение:** Уведомление получателя о прибытии международного почтового отправления в почтовое отделение; содержит инструкции по получению и сумму к оплате (если есть пошлина).

**Пользователь:** Оператор (формирует), Получатель (получает).

**Структура извещения:**

**Заголовок:**

| Поле | Источник |
|------|---------|
| Наименование почтового отделения | Справочник учреждения |
| Адрес отделения, режим работы | Справочник учреждения |
| Название: «ИЗВЕЩЕНИЕ О ПОСТУПЛЕНИИ МЕЖДУНАРОДНОГО ПОЧТОВОГО ОТПРАВЛЕНИЯ» | Константа |
| Внутренний номер / Трек-номер | Пересылка |
| Дата оформления извещения | Текущая дата |

**Блок данных отправления:**

| Поле | Источник |
|------|---------|
| Трек-номер | Пересылка |
| Страна отправления | Пересылка |
| Наименование отправителя | Пересылка |
| Вид отправления | Пересылка |
| Объявленная стоимость | Пересылка |
| Вес | Пересылка |
| Дата поступления в отделение | Пересылка |
| Срок хранения до | Дата поступления + N суток (из настроек) |

**Блок получателя:**

| Поле | Источник |
|------|---------|
| ФИО получателя | Пересылка |
| Адрес получателя | Пересылка |

**Блок платёжной информации (отображается только при пошлине > 0):**

| Поле | Источник |
|------|---------|
| Таможенная пошлина (сумма к оплате) | Рассчитано системой / скорректировано оператором |
| Сбор за хранение (при наличии) | Настройки |
| Итого к оплате | Сумма полей |
| Реквизиты для оплаты | Справочник учреждения |

**Блок инструкций:**
```
Для получения отправления необходимо:
1. Оплатить сумму к оплате (при наличии).
2. Предъявить данное извещение.
3. Предъявить документ, удостоверяющий личность.
Отправление хранится до: [дата срока хранения]
```

**Подвал:**

| Поле | Источник |
|------|---------|
| ФИО оператора | Текущий пользователь |
| Дата оформления | Автоматически |
| Печать отделения | Изображение из настроек |

**Формат вывода:** PDF, A5 (сложенный A4) или A4. Кириллица. Дополнительно — отображение в браузере.

## 4. Приложение — Диаграмма вариантов использования

```
Актёры:
  [Оператор]  [Старший оператор]  [Администратор]  [Получатель*]

Use Cases:
  UC-01: Регистрация входящей пересылки         ← Оператор
  UC-02: Поиск и просмотр пересылки             ← Оператор, Старший оператор
  UC-03: Расчёт и учёт пошлины                  ← Оператор (авто + ручная корр.)
  UC-04: Фиксация оплаты пошлины                ← Оператор
  UC-05: Печать извещения получателю            ← Оператор
  UC-06: Регистрация выдачи отправления         ← Оператор
  UC-07: Учёт невостребованных (возврат)        ← Система (авто) + Оператор
  UC-08: Формирование отчётов                   ← Старший оператор
  UC-09: Управление настройками (пошлины, сроки)← Администратор
  UC-10: Управление пользователями              ← Администратор

* Уведомление получателя через СМС/email — вне данной версии
```
