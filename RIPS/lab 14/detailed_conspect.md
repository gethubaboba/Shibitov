# Подробный конспект по лабораторной работе 14

## 1. Что требовалось сделать

В лабораторной работе нужно было реализовать клиент-серверное приложение на основе сокетов.
По условию есть распределенная система хранения данных. Она состоит не из одного сервера,
а из нескольких узлов. Каждый узел хранит одну пару:

```text
ключ -> значение
```

Например:

```text
alpha -> red
beta  -> green
gamma -> blue
delta -> yellow
omega -> white
```

Клиент не обязан знать, на каком именно узле лежит нужный ключ. Он может подключиться
к любому узлу системы и спросить:

```text
дай значение по ключу omega
```

Если узел сам хранит этот ключ, он сразу отвечает клиенту. Если не хранит, он должен
попробовать найти ключ у соседних узлов. То есть система работает как маленькая сеть
серверов, которые умеют пересылать запросы друг другу.

В реализации использован язык Java и обычные TCP-сокеты:

- `ServerSocket` - серверный сокет, который слушает порт;
- `Socket` - соединение между клиентом и сервером или между двумя серверами;
- текстовые команды - простой протокол общения между участниками.

## 2. Главная идея реализации

Реализация состоит из четырех Java-файлов:

```text
src/NodeServer.java
src/KeyValueClient.java
src/Topology.java
src/DemoLauncher.java
```

Каждый файл отвечает за свою часть:

- `NodeServer.java` - один узел распределенного хранилища.
- `KeyValueClient.java` - клиент, который отправляет запрос `GET`.
- `Topology.java` - построение связей между узлами.
- `DemoLauncher.java` - демонстрационный запуск нескольких узлов сразу.

Если говорить совсем просто:

```text
KeyValueClient  ->  NodeServer  ->  другие NodeServer
```

Клиент спрашивает один узел. Этот узел либо отвечает сам, либо спрашивает соседей.

## 3. Что такое узел системы

Узел - это отдельный сервер. В коде он представлен классом `NodeServer`.

У каждого узла есть:

```java
private final int port;
private final String key;
private final String value;
private final List<Integer> peers;
```

Разберем каждое поле.

### 3.1. `port`

`port` - это номер порта, на котором работает сервер.

Например, если узел работает на порту `5201`, то клиент может подключиться к нему так:

```text
localhost:5201
```

В демонстрации используются пять портов:

```text
5201, 5202, 5203, 5204, 5205
```

Важно: каждый сервер должен слушать свой порт. Если два сервера попытаются занять один
и тот же порт, второй не запустится.

### 3.2. `key` и `value`

`key` и `value` - это локально хранимая пара данных.

Например:

```java
new NodeServer(5201, "alpha", "red", peers)
```

Это означает:

```text
узел 5201 хранит alpha=red
```

По условию лабораторной на каждом узле хранится одна пара `ключ-значение`, поэтому
обычная база данных здесь не нужна. Достаточно двух строковых полей.

### 3.3. `peers`

`peers` - это список соседних узлов. В реализации соседи задаются через номера портов.

Например:

```text
узел 5201 имеет соседей [5202, 5203]
```

Это значит, что если узел `5201` не нашел ключ у себя, он может обратиться к узлам
`5202` и `5203`.

Список соседей зависит от выбранной топологии:

- в полносвязной топологии каждый узел связан со всеми остальными;
- в звезде все узлы связаны с центральным узлом;
- в дереве узлы соединены как бинарное дерево.

## 4. Как запускается серверный узел

Запуск узла происходит в методе `start()`:

```java
public void start() throws IOException {
    serverSocket = new ServerSocket(port);
    serverSocket.setSoTimeout(250);
    running.set(true);
    acceptThread = new Thread(this::acceptLoop, "node-" + port);
    acceptThread.start();
    System.out.printf("Node %d started: %s=%s, peers=%s%n", port, key, value, peers);
}
```

Пошагово:

1. Создается `ServerSocket`.
2. Сервер начинает слушать указанный порт.
3. Флаг `running` переводится в `true`.
4. Создается отдельный поток `acceptThread`.
5. Этот поток запускает цикл приема подключений `acceptLoop()`.

Почему прием подключений вынесен в отдельный поток?

Потому что метод `accept()` блокирующий. Это значит: если сервер вызвал `accept()`,
он будет ждать клиента и не пойдет дальше, пока кто-нибудь не подключится. Если бы
мы запускали `accept()` прямо в основном потоке, программа застряла бы на первом
сервере и не смогла бы запустить остальные узлы в `DemoLauncher`.

## 5. Что делает `acceptLoop()`

Метод `acceptLoop()` постоянно ждет новые подключения:

```java
private void acceptLoop() {
    while (running.get()) {
        try {
            Socket socket = serverSocket.accept();
            workers.submit(() -> handle(socket));
        } catch (SocketTimeoutException ignored) {
            // Allows the loop to observe the running flag.
        } catch (IOException e) {
            if (running.get()) {
                System.out.printf("Node %d accept error: %s%n", port, e.getMessage());
            }
        }
    }
}
```

Главная строка:

```java
Socket socket = serverSocket.accept();
```

Она означает:

```text
ждать, пока кто-нибудь подключится
```

Когда подключение появилось, сервер получает объект `Socket`. Через него можно читать
запрос и отправлять ответ.

После этого соединение передается в пул потоков:

```java
workers.submit(() -> handle(socket));
```

Пул потоков нужен, чтобы сервер мог обслуживать несколько клиентов. Если один клиент
подключился и сервер с ним работает, другой клиент не обязан ждать завершения всей
программы. Его запрос может быть обработан отдельным рабочим потоком.

## 6. Зачем нужен `setSoTimeout(250)`

В методе `start()` есть строка:

```java
serverSocket.setSoTimeout(250);
```

Она задает таймаут для `accept()`. Без таймаута метод `accept()` мог бы ждать клиента
бесконечно. Это создало бы проблему при остановке сервера.

Например, метод `close()` ставит:

```java
running.set(false);
```

Но если поток застрял внутри `accept()`, он не сможет сразу проверить новый флаг.

Таймаут `250` миллисекунд решает это так:

```text
accept() ждет клиента максимум 250 мс,
если клиента нет - бросает SocketTimeoutException,
цикл снова проверяет running,
если running=false - поток завершается.
```

То есть таймаут нужен для корректной остановки сервера.

## 7. Как сервер читает запрос клиента

За обработку одного соединения отвечает метод `handle()`:

```java
private void handle(Socket socket) {
    try (socket;
         BufferedReader in = new BufferedReader(new InputStreamReader(socket.getInputStream()));
         PrintWriter out = new PrintWriter(socket.getOutputStream(), true)) {
        String request = in.readLine();
        out.println(process(request));
    } catch (IOException e) {
        System.out.printf("Node %d client error: %s%n", port, e.getMessage());
    }
}
```

Здесь создаются два объекта:

```java
BufferedReader in
PrintWriter out
```

`BufferedReader` нужен для чтения текста из сокета.  
`PrintWriter` нужен для отправки текста обратно.

Вся коммуникация сделана строками. Клиент отправляет одну строку, сервер отправляет
одну строку ответа.

Пример клиентского запроса:

```text
GET omega
```

Пример ответа:

```text
FOUND omega=white at node 5205
```

Или, если ключ не найден:

```text
NOT_FOUND missing
```

## 8. Почему используется `try-with-resources`

Метод `handle()` написан так:

```java
try (socket;
     BufferedReader in = ...;
     PrintWriter out = ...) {
    ...
}
```

Это называется `try-with-resources`. Он автоматически закрывает ресурсы после выхода
из блока. То есть сокет, поток чтения и поток записи будут закрыты даже в случае ошибки.

Это важно, потому что незакрытые сокеты могут занимать системные ресурсы и мешать
повторному запуску программы.

## 9. Как устроен протокол команд

Протокол - это правила, по которым клиент и сервер понимают друг друга.

В лабораторной используется простой текстовый протокол:

```text
GET <key>
PEER_GET <key> <visitedPorts>
```

### 9.1. Команда `GET`

Команда `GET` используется обычным клиентом.

Пример:

```text
GET alpha
```

Смысл:

```text
клиент просит найти значение по ключу alpha
```

Если клиент подключился к узлу `5201`, а узел `5201` хранит `alpha=red`, ответ будет:

```text
FOUND alpha=red at node 5201
```

### 9.2. Команда `PEER_GET`

Команда `PEER_GET` используется не обычным клиентом, а одним сервером при обращении
к другому серверу.

Пример:

```text
PEER_GET omega 5201,5202
```

Смысл:

```text
найди ключ omega, но знай, что узлы 5201 и 5202 уже были посещены
```

Список посещенных узлов нужен, чтобы запрос не ходил по кругу бесконечно.

## 10. Как сервер разбирает запрос

Разбор запроса выполняется в методе `process()`:

```java
private String process(String request) {
    if (request == null || request.isBlank()) {
        return "ERROR empty request";
    }

    String[] parts = request.trim().split("\\s+");
    if (parts.length == 2 && parts[0].equals("GET")) {
        return find(parts[1], new HashSet<>(Set.of(port)));
    }
    if (parts.length >= 3 && parts[0].equals("PEER_GET")) {
        Set<Integer> visited = parseVisited(parts[2]);
        visited.add(port);
        return find(parts[1], visited);
    }

    return "ERROR unknown command";
}
```

Пояснение:

1. Если запрос пустой, возвращается ошибка.
2. Запрос разбивается на слова через пробелы.
3. Если первое слово `GET`, значит запрос пришел от клиента.
4. Если первое слово `PEER_GET`, значит запрос пришел от другого узла.
5. Для обеих команд вызывается метод `find()`.

Для обычного `GET` создается начальный список посещенных узлов:

```java
new HashSet<>(Set.of(port))
```

То есть текущий узел сразу считается посещенным.

## 11. Главный алгоритм поиска ключа

Главная логика находится в методе `find()`:

```java
private String find(String requestedKey, Set<Integer> visited) {
    if (key.equals(requestedKey)) {
        return "FOUND " + key + "=" + value + " at node " + port;
    }

    for (int peer : peers) {
        if (visited.contains(peer)) {
            continue;
        }

        Set<Integer> nextVisited = new HashSet<>(visited);
        nextVisited.add(peer);
        String response = askPeer(peer, requestedKey, nextVisited);
        if (response != null && response.startsWith("FOUND")) {
            return response;
        }
    }

    return "NOT_FOUND " + requestedKey;
}
```

Разберем по шагам.

### Шаг 1. Проверка локального ключа

```java
if (key.equals(requestedKey)) {
    return "FOUND " + key + "=" + value + " at node " + port;
}
```

Если текущий узел сам хранит нужный ключ, больше ничего делать не нужно. Ответ сразу
возвращается клиенту или предыдущему узлу.

Например, текущий узел:

```text
5205: omega=white
```

Запрос:

```text
GET omega
```

Ответ:

```text
FOUND omega=white at node 5205
```

### Шаг 2. Обход соседей

Если локально ключ не найден, сервер проходит по соседям:

```java
for (int peer : peers) {
    ...
}
```

Каждый `peer` - это порт соседнего узла.

### Шаг 3. Пропуск уже посещенных узлов

```java
if (visited.contains(peer)) {
    continue;
}
```

Если сосед уже был посещен, повторно идти туда нельзя.

Почему?

Представим полносвязную топологию:

```text
5201 связан с 5202
5202 связан с 5201
```

Если не помнить посещенные узлы, может возникнуть бесконечная пересылка:

```text
5201 -> 5202 -> 5201 -> 5202 -> ...
```

Поэтому список `visited` защищает систему от циклов.

### Шаг 4. Отправка запроса соседу

```java
String response = askPeer(peer, requestedKey, nextVisited);
```

Если сосед нашел ключ, он вернет строку, начинающуюся с `FOUND`.

Тогда текущий узел сразу возвращает этот ответ выше:

```java
if (response != null && response.startsWith("FOUND")) {
    return response;
}
```

### Шаг 5. Если никто не нашел ключ

Если все соседи проверены, а ключ не найден:

```java
return "NOT_FOUND " + requestedKey;
```

## 12. Как узел обращается к соседу

Для обращения к соседнему узлу используется метод `askPeer()`:

```java
private String askPeer(int peerPort, String requestedKey, Set<Integer> visited) {
    try (Socket socket = new Socket("localhost", peerPort);
         BufferedReader in = new BufferedReader(new InputStreamReader(socket.getInputStream()));
         PrintWriter out = new PrintWriter(socket.getOutputStream(), true)) {
        out.println("PEER_GET " + requestedKey + " " + serializeVisited(visited));
        return in.readLine();
    } catch (IOException e) {
        return "NOT_FOUND " + requestedKey;
    }
}
```

Здесь сервер сам становится клиентом для другого сервера.

Например, узел `5201` не нашел `omega` у себя. Он обращается к соседу `5202`:

```java
new Socket("localhost", 5202)
```

И отправляет:

```text
PEER_GET omega 5201,5202
```

Почему используется `localhost`?

Потому что демонстрация запускает все узлы на одном компьютере. В реальной сети вместо
`localhost` можно было бы хранить адрес каждого узла, например `192.168.1.10`.

## 13. Как хранится список посещенных узлов

Список посещенных узлов хранится как `Set<Integer>`:

```java
Set<Integer> visited
```

`Set` - это множество. В множестве не бывает одинаковых элементов.

Это удобно, потому что один и тот же порт не должен повторяться несколько раз.

Для передачи по сети множество превращается в строку:

```text
5201,5202,5203
```

За это отвечает метод `serializeVisited()`:

```java
private static String serializeVisited(Set<Integer> visited) {
    List<Integer> ordered = new ArrayList<>(visited);
    ordered.sort(Integer::compareTo);
    StringBuilder sb = new StringBuilder();
    for (int i = 0; i < ordered.size(); i++) {
        if (i > 0) {
            sb.append(',');
        }
        sb.append(ordered.get(i));
    }
    return sb.toString();
}
```

Почему сначала создается список и сортируется?

Чтобы строка была предсказуемой:

```text
5201,5202,5203
```

а не случайный порядок элементов множества.

Обратное преобразование делает `parseVisited()`:

```java
private static Set<Integer> parseVisited(String text) {
    Set<Integer> visited = new HashSet<>();
    for (String item : text.split(",")) {
        if (!item.isBlank()) {
            visited.add(Integer.parseInt(item));
        }
    }
    return visited;
}
```

Он берет строку:

```text
5201,5202
```

и превращает ее обратно в множество:

```text
{5201, 5202}
```

## 14. Как работает клиент

Клиент реализован в файле `KeyValueClient.java`.

Главный метод:

```java
public static String get(String host, int port, String key) throws Exception {
    try (Socket socket = new Socket(host, port);
         BufferedReader in = new BufferedReader(new InputStreamReader(socket.getInputStream()));
         PrintWriter out = new PrintWriter(socket.getOutputStream(), true)) {
        out.println("GET " + key);
        return in.readLine();
    }
}
```

Пошагово:

1. Клиент открывает сокет к серверу.
2. Отправляет строку `GET <key>`.
3. Ждет одну строку ответа.
4. Возвращает ответ.

Пример ручного запуска:

```powershell
java -cp build KeyValueClient localhost 5201 omega
```

Это значит:

```text
подключись к localhost:5201 и спроси ключ omega
```

## 15. Что делает `Topology.java`

Файл `Topology.java` отвечает за построение связей между узлами.

В нем есть перечисление:

```java
public enum Topology {
    FULL,
    STAR,
    TREE;
}
```

`enum` - это набор заранее известных вариантов.

В лабораторной поддерживаются три топологии:

```text
FULL - полносвязная
STAR - звезда
TREE - дерево
```

Метод:

```java
public List<List<Integer>> buildPeerPorts(List<Integer> ports)
```

получает список портов:

```text
[5201, 5202, 5203, 5204, 5205]
```

и возвращает список соседей для каждого узла.

## 16. Полносвязная топология

Полносвязная топология означает:

```text
каждый узел связан с каждым другим узлом
```

Для пяти узлов это выглядит так:

```text
5201 -> 5202, 5203, 5204, 5205
5202 -> 5201, 5203, 5204, 5205
5203 -> 5201, 5202, 5204, 5205
5204 -> 5201, 5202, 5203, 5205
5205 -> 5201, 5202, 5203, 5204
```

В коде:

```java
for (int i = 0; i < ports.size(); i++) {
    for (int j = 0; j < ports.size(); j++) {
        if (i != j) {
            peers.get(i).add(ports.get(j));
        }
    }
}
```

Здесь двойной цикл перебирает все пары узлов. Проверка `i != j` нужна, чтобы узел
не добавлял в соседи сам себя.

## 17. Топология звезда

В звезде есть центральный узел. В реализации центральным считается первый порт:

```text
5201
```

Связи:

```text
5201 -> 5202, 5203, 5204, 5205
5202 -> 5201
5203 -> 5201
5204 -> 5201
5205 -> 5201
```

То есть все боковые узлы знают только центр, а центр знает всех.

В коде:

```java
for (int i = 1; i < ports.size(); i++) {
    peers.get(0).add(ports.get(i));
    peers.get(i).add(ports.get(0));
}
```

`peers.get(0)` - это список соседей центрального узла.  
`peers.get(i)` - список соседей очередного бокового узла.

## 18. Топология дерево

Дерево реализовано как бинарное дерево.

Для узла с индексом `i`:

```text
левый потомок  = 2 * i + 1
правый потомок = 2 * i + 2
```

Например, для пяти узлов:

```text
индекс 0: порт 5201
индекс 1: порт 5202
индекс 2: порт 5203
индекс 3: порт 5204
индекс 4: порт 5205
```

Связи:

```text
5201 -> 5202, 5203
5202 -> 5201, 5204, 5205
5203 -> 5201
5204 -> 5202
5205 -> 5202
```

В коде:

```java
int left = 2 * i + 1;
int right = 2 * i + 2;
if (left < ports.size()) {
    link(peers, ports, i, left);
}
if (right < ports.size()) {
    link(peers, ports, i, right);
}
```

Метод `link()` добавляет связь в обе стороны:

```java
peers.get(a).add(ports.get(b));
peers.get(b).add(ports.get(a));
```

Это важно: если родитель знает ребенка, ребенок тоже должен знать родителя.

## 19. Как работает демонстрационный запуск

Демонстрация находится в `DemoLauncher.java`.

В ней создаются пять узлов:

```java
List<Integer> ports = List.of(5201, 5202, 5203, 5204, 5205);
List<String> keys = List.of("alpha", "beta", "gamma", "delta", "omega");
List<String> values = List.of("red", "green", "blue", "yellow", "white");
```

Получается такая таблица:

```text
5201: alpha=red
5202: beta=green
5203: gamma=blue
5204: delta=yellow
5205: omega=white
```

Дальше выбирается топология:

```java
Topology topology = args.length == 0 ? Topology.FULL : Topology.fromName(args[0]);
```

Если аргумент не передан, используется `FULL`.  
Если передан аргумент, например `star`, используется топология `STAR`.

Примеры:

```powershell
java -cp build DemoLauncher full
java -cp build DemoLauncher star
java -cp build DemoLauncher tree
```

## 20. Почему `DemoLauncher` удобен

Без `DemoLauncher` пришлось бы запускать каждый узел в отдельном терминале:

```powershell
java -cp build NodeServer 5201 alpha red 5202,5203
java -cp build NodeServer 5202 beta green 5201
...
```

Это неудобно для проверки лабораторной.

`DemoLauncher` запускает все узлы внутри одного процесса:

```java
for (int i = 0; i < ports.size(); i++) {
    NodeServer node = new NodeServer(ports.get(i), keys.get(i), values.get(i), peers.get(i));
    node.start();
    nodes.add(node);
}
```

Затем делает несколько клиентских запросов:

```java
request(ports.get(0), "alpha");
request(ports.get(0), "omega");
request(ports.get(2), "delta");
request(ports.get(4), "missing");
```

Эти запросы проверяют разные ситуации:

- ключ найден на том же узле;
- ключ найден на удаленном узле;
- запрос пришел не в центральный узел, но все равно нашел данные;
- ключ отсутствует во всей системе.

## 21. Пример полного прохождения запроса

Рассмотрим запрос:

```text
Client -> node 5201, key=omega
```

Допустим используется топология `TREE`.

Данные:

```text
5201: alpha=red
5202: beta=green
5203: gamma=blue
5204: delta=yellow
5205: omega=white
```

Связи дерева:

```text
5201 -> 5202, 5203
5202 -> 5201, 5204, 5205
5203 -> 5201
5204 -> 5202
5205 -> 5202
```

Путь поиска:

```text
1. Клиент отправляет GET omega на 5201.
2. Узел 5201 проверяет себя: у него alpha, значит omega нет.
3. Узел 5201 спрашивает соседа 5202.
4. Узел 5202 проверяет себя: у него beta, значит omega нет.
5. Узел 5202 не идет обратно на 5201, потому что 5201 уже в visited.
6. Узел 5202 спрашивает 5204.
7. Узел 5204 проверяет себя: у него delta, значит omega нет.
8. Узел 5204 не идет обратно на 5202, потому что 5202 уже посещен.
9. Узел 5204 возвращает NOT_FOUND.
10. Узел 5202 спрашивает 5205.
11. Узел 5205 проверяет себя: у него omega=white.
12. Узел 5205 возвращает FOUND omega=white at node 5205.
13. Узел 5202 передает этот ответ назад.
14. Узел 5201 передает этот ответ клиенту.
```

Итоговый ответ:

```text
FOUND omega=white at node 5205
```

## 22. Как программа останавливает серверы

`NodeServer` реализует `AutoCloseable`:

```java
public class NodeServer implements AutoCloseable
```

Это значит, что у него есть метод `close()`.

В `close()` происходит:

```java
running.set(false);
serverSocket.close();
acceptThread.join(1000);
workers.shutdownNow();
```

Пошагово:

1. `running=false` говорит циклу приема подключений остановиться.
2. `serverSocket.close()` закрывает серверный сокет.
3. `acceptThread.join(1000)` ждет завершения потока приема подключений.
4. `workers.shutdownNow()` останавливает пул рабочих потоков.

В `DemoLauncher` остановка находится в блоке `finally`:

```java
finally {
    for (NodeServer node : nodes) {
        node.close();
    }
}
```

`finally` выполнится даже если внутри демонстрации возникнет ошибка. Это защищает
от ситуации, когда серверы остались висеть и занимать порты.

Потом можно запустить несколько серверов в разных терминалах.


## 24. Почему реализация считается распределенной

Система называется распределенной не потому, что обязательно запущена на разных
физических компьютерах. Главное, что данные не лежат в одном общем объекте.

В этой реализации:

- каждый узел имеет собственный серверный сокет;
- каждый узел хранит только свою пару `ключ-значение`;
- узлы общаются между собой через сетевые соединения;
- клиент может подключиться к любому узлу;
- результат может быть найден на другом узле.

Даже если все узлы запущены на одном компьютере через `localhost`, архитектурно они
работают как отдельные сетевые участники.

## 25. Ограничения реализации

Реализация учебная, поэтому она специально простая.

Ограничения:

- данные хранятся только в памяти;
- после остановки программы данные исчезают;
- каждый узел хранит только одну пару;
- адрес соседа считается `localhost`;
- пересылка запросов синхронная: узел ждет ответ соседа;
- нет авторизации;
- нет шифрования;
- нет репликации;
- нет балансировки нагрузки;
- нет обработки сложных сетевых сбоев.

Для лабораторной это нормально, потому что цель работы - показать клиент-серверное
взаимодействие, сокеты, топологии и пересылку запросов между узлами.

## 26. Что можно было бы улучшить

Если развивать эту систему дальше, можно добавить:

- хранение нескольких пар на одном узле через `Map<String, String>`;
- чтение конфигурации узлов из JSON или YAML;
- реальные IP-адреса вместо `localhost`;
- таймауты на клиентские запросы к соседям;
- логирование через `java.util.logging`;
- репликацию данных на несколько узлов;
- удаление и добавление ключей;
- команды `PUT`, `DELETE`, `LIST`;
- асинхронный поиск сразу по нескольким соседям;
- HTTP или gRPC вместо самодельного текстового протокола.

## 27. Ответы на возможные вопросы

### Вопрос 1. Почему выбран Java, а не C++?

Java хорошо подходит для такой лабораторной: в стандартной библиотеке уже есть
`ServerSocket`, `Socket`, потоки, коллекции и удобная работа со строками. Поэтому
можно сосредоточиться на логике распределенного хранилища, а не на низкоуровневой
работе с памятью.

### Вопрос 2. Почему каждый узел хранит только одну пару?

Так сказано в условии лабораторной: на каждом узле хранится одна пара в формате
`ключ-значение`. Поэтому реализация использует поля `key` и `value`, а не полноценную
таблицу или базу данных.

### Вопрос 3. Почему используются порты как идентификаторы узлов?

В демонстрации все узлы работают на одном компьютере. Чтобы отличать их друг от друга,
удобно использовать разные порты. Порт одновременно является и адресом подключения,
и простым идентификатором узла.

### Вопрос 4. Что будет, если ключ находится на первом же узле?

Тогда поиск не идет к соседям. Метод `find()` сразу вернет `FOUND`, потому что сначала
проверяется локальная пара `key=value`.

### Вопрос 5. Что будет, если ключа нет нигде?

Запрос обойдет доступные узлы согласно топологии. Когда все возможные соседи будут
проверены, вернется:

```text
NOT_FOUND <key>
```

### Вопрос 6. Почему нужен список `visited`?

Он нужен для защиты от циклов. В топологиях `full` и `star` есть пути, по которым
запрос мог бы вернуться назад. Без `visited` серверы могли бы пересылать запрос друг
другу бесконечно.

### Вопрос 7. Почему `visited` передается строкой?

Потому что протокол текстовый. Сокеты передают байты, а программа договорилась
обмениваться строками. Поэтому множество портов превращается в строку вида:

```text
5201,5202,5203
```

А на принимающей стороне строка снова превращается в `Set<Integer>`.

### Вопрос 8. Почему в `askPeer()` при ошибке возвращается `NOT_FOUND`?

Для учебной реализации это простой способ продолжить работу. Если сосед недоступен,
текущий узел считает, что через этого соседа ключ не найден. В промышленной системе
лучше было бы возвращать отдельную ошибку, например `PEER_UNAVAILABLE`.

### Вопрос 9. Может ли сервер обслужить несколько клиентов одновременно?

Да. Для этого используется:

```java
ExecutorService workers = Executors.newCachedThreadPool();
```

Каждое новое подключение передается в пул потоков:

```java
workers.submit(() -> handle(socket));
```

Поэтому несколько запросов могут обрабатываться параллельно.

### Вопрос 10. Почему `PrintWriter` создан с параметром `true`?

Конструктор:

```java
new PrintWriter(socket.getOutputStream(), true)
```

включает автоматический сброс буфера после `println()`. Это важно: если данные
останутся в буфере, другая сторона может не получить ответ сразу и будет ждать.

### Вопрос 11. Почему используется `readLine()`?

Протокол построен так, что один запрос - это одна строка, и один ответ - тоже одна
строка. Поэтому `readLine()` идеально подходит: он читает текст до конца строки.

### Вопрос 12. Что означает `localhost`?

`localhost` - это текущий компьютер. В демонстрации все серверы запущены на одной
машине, но на разных портах. Поэтому подключение идет к `localhost:5201`,
`localhost:5202` и так далее.

### Вопрос 13. Можно ли запустить узлы на разных компьютерах?

Да, но код нужно немного расширить. Сейчас соседи описаны только портами, а адрес
жестко задан как `localhost`. Для реальной сети нужно хранить пару:

```text
host + port
```

Например:

```text
192.168.1.10:5201
192.168.1.11:5202
```

### Вопрос 14. Чем отличаются топологии?

Полносвязная топология дает самый короткий путь между узлами, но требует больше
соединений. Звезда проще: все идет через центр. Дерево уменьшает количество связей,
но путь до ключа может быть длиннее.

### Вопрос 15. Почему в дереве используется формула `2*i+1` и `2*i+2`?

Это стандартный способ хранить бинарное дерево в массиве или списке. Для узла с
индексом `i` левый ребенок имеет индекс `2*i+1`, правый - `2*i+2`.

### Вопрос 16. Почему `NodeServer` реализует `AutoCloseable`?

Так у сервера появляется стандартный метод `close()`. Это удобно для освобождения
ресурсов: закрытия сокета, остановки потоков и завершения пула работников.

### Вопрос 17. Что делает `shutdownHook` в `main()` класса `NodeServer`?

Он добавляет обработчик завершения программы. Если сервер запускается вручную и
пользователь остановит процесс, Java попытается вызвать `server.close()` и аккуратно
освободить ресурсы.

### Вопрос 18. Почему в `DemoLauncher` есть `Thread.sleep(500)`?

После запуска серверов им нужно немного времени, чтобы открыть порты и начать слушать
подключения. Пауза 500 мс делает демонстрацию стабильнее: клиенты начинают запросы
уже после старта серверов.

### Вопрос 19. Что будет, если порт уже занят?

`new ServerSocket(port)` выбросит исключение `IOException`. Это означает, что сервер
не сможет стартовать на этом порту. Нужно выбрать другой свободный порт или остановить
процесс, который уже занимает этот порт.

### Вопрос 20. Почему данные не сохраняются в файл?

По условию нужно показать распределенную систему хранения и работу с сокетами.
Постоянное хранение на диске не является главной целью. Для простоты пары хранятся
в оперативной памяти.

### Вопрос 21. Почему в ответе указывается номер узла?

Ответ:

```text
FOUND omega=white at node 5205
```

показывает не только значение, но и где оно было найдено. Это удобно для проверки:
сразу видно, что запрос действительно дошел до удаленного узла.

### Вопрос 22. Почему нет команды `PUT`?

В задании требуется запрашивать данные по ключу, а не изменять их. Поэтому реализована
только команда чтения `GET`. Команду `PUT` можно добавить как развитие проекта.

### Вопрос 23. Почему в full-топологии все равно нужен обход, если каждый узел знает всех?

Даже если узел знает всех, он не знает заранее, у кого именно нужный ключ. Поэтому он
последовательно спрашивает соседей, пока не получит `FOUND` или пока соседи не закончатся.

### Вопрос 24. Есть ли здесь настоящая многопоточность?

Да. У каждого `NodeServer` есть отдельный поток приема подключений. Кроме того,
каждое клиентское соединение обрабатывается в пуле потоков `ExecutorService`.

### Вопрос 25. Почему эта реализация подходит для лабораторной?

Она покрывает основные требования:

- есть серверные узлы;
- каждый узел хранит свою пару `ключ-значение`;
- клиент может подключиться к любому узлу;
- запрос может быть найден на другом узле;
- реализованы разные топологии;
- результаты запросов выводятся у клиента;
- проект можно собрать и проверить стандартными командами Java.
