## 2. Задание 1–2. Этапы разработки параллельных алгоритмов

### 2.1. Методика PCAM (Гергель, §3.2)

Методика состоит из четырёх последовательных этапов:

| Этап | Название | Содержание |
|:----:|----------|------------|
| **P** | **Разбиение** (Partitioning) | Декомпозиция задачи на множество независимых подзадач. Два вида: *функциональное* (разбиение вычислений) и *по данным* (разбиение данных, из него выводятся вычисления). |
| **C** | **Коммуникация** (Communication) | Определение обменов данными между подзадачами: локальные / глобальные; синхронные / асинхронные. |
| **A** | **Агрегирование** (Agglomeration) | Объединение мелких подзадач в более крупные для сокращения накладных расходов на коммуникацию и управление. |
| **M** | **Отображение** (Mapping) | Назначение агрегированных задач процессорным узлам; балансировка нагрузки. |

**Критерии оценки корректности каждого этапа:**

- **P:** достаточно ли подзадач? Нет ли избыточных зависимостей? Масштабируется ли разбиение?
- **C:** разумны ли объёмы коммуникации? Нет ли глобальных рассылок там, где достаточно локальных обменов?
- **A:** снизилось ли число коммуникаций? Сохранена ли гибкость масштабирования?
- **M:** сбалансирована ли нагрузка? Минимизирована ли стоимость коммуникации?

### 2.2. Гравитационная задача N тел (Гергель, §3.3) — разбор примера

**Задача:** N тел с заданными координатами, массами и начальными скоростями. На каждом шаге моделирования вычислить силы притяжения между всеми парами тел и обновить координаты.

| Этап | Применение |
|------|-----------|
| **P** | Разбиение по данным: каждому телу `i` соответствует подзадача вычисления суммарной силы от всех остальных тел. Итого N подзадач. |
| **C** | Каждая подзадача нуждается в координатах и массах всех N тел → глобальная рассылка (broadcast) на каждом шаге. |
| **A** | Объединяем несколько тел на один процессор; число коммуникаций не меняется (broadcast), но уменьшается число процессов. |
| **M** | Статическое равномерное распределение: процессор `p` обрабатывает тела с `p*N/P` по `(p+1)*N/P`. |

## 3. Задание 3. Умножение матрицы на вектор: схема параллельных вычислений

**Задача:** Вычислить **y = A · x**, где A — матрица размера n×n, x — вектор длины n, y — вектор-результат длины n.

Каждый элемент результата:
```
y[i] = A[i][0]*x[0] + A[i][1]*x[1] + ... + A[i][n-1]*x[n-1]   (i = 0..n-1)
```

### 3.1. Этап P — Разбиение

**Вид разбиения:** по данным (data decomposition) — строкам матрицы A.

- Подзадача `k` вычисляет y[k] = скалярное произведение k-й строки A на вектор x.
- Итого: **n подзадач**, каждая состоит из n умножений и n-1 сложений.
- Подзадачи полностью независимы между собой (нет зависимостей по данным).

```
Подзадача 0:  y[0] = A[0][:] · x    (n операций)
Подзадача 1:  y[1] = A[1][:] · x    (n операций)
...
Подзадача n-1: y[n-1] = A[n-1][:] · x
```

**Оценка P:** подзадачи равноёмкие (каждая n операций) — разбиение сбалансировано. Количество подзадач n при n=10000 избыточно велико → потребуется агрегирование.

### 3.2. Этап C — Коммуникация

- Вектор **x** (размером n) необходим **всем** подзадачам → **глобальная рассылка** (broadcast) от узла, хранящего x, ко всем.
- Строки матрицы A распределяются по процессорам (каждый хранит «свои» строки) → дополнительной коммуникации не требуется.
- После вычислений y[k] возвращаются на главный узел → **сбор** (gather) результатов.

| Коммуникация | Тип | Объём |
|-------------|-----|-------|
| Рассылка x | broadcast | n × sizeof(double) |
| Распределение строк A | scatter | n²/P × sizeof(double) на процессор |
| Сбор результата y | gather | n × sizeof(double) |

**Оценка C:** объём коммуникации O(n²/P + n) — при большом n доминирует рассылка x (O(n)), а передача строк матрицы O(n²/P) сопоставима с вычислениями O(n²/P) → коммуникация эффективна.

### 3.3. Этап A — Агрегирование

При наличии P процессоров объединяем n/P строк в одну задачу:

- **Задача процессора p** вычисляет блок строк A[p·n/P .. (p+1)·n/P - 1], т.е. n/P элементов y.
- Каждый процессор получает одну копию x (broadcast) и n/P строк A.
- Число коммуникационных операций сокращается с n до P.

**Оценка A:** при P=8, n=4000: было 4000 мелких подзадач → 8 крупных. Коммуникация: передача блока A размером n²/P = 2·10⁶ элементов + рассылка x = 4000 элементов. Гранулярность высокая — агрегирование корректно.

### 3.4. Этап M — Отображение

**Статическое** равномерное распределение строк (поскольку все задачи равноёмкие):

```
Процессор p: строки [p * n/P, (p+1) * n/P)
```

**Оценка M:** нагрузка сбалансирована (равное число строк), топология коммуникации «звезда» (broadcast от корневого процессора) — минимальная стоимость. Отображение оптимально.

### 3.5. Оценка эффективности

| Параметр | Формула | При P=8, n=4000 |
|----------|---------|-----------------|
| Время вычислений | T_comp = 2n²/P операций | ~4·10⁶ оп. |
| Время коммуникации | T_comm ≈ α + β·n (broadcast) | пренебрежимо |
| Ускорение теоретическое | S ≈ P = 8 | ~8× |
| КПД | E = S/P | ~1 (идеально) |

**Задача умножения матрицы на вектор масштабируется практически линейно** при равномерном разбиении по строкам, поскольку последовательный участок (broadcast x) имеет объём O(n) против O(n²/P) вычислений.

## 4. Задание 4. Подсчёт вхождений слова в файлах: схема параллельных вычислений

**Задача:** дан список файлов F = {f₀, f₁, ..., f_{m-1}} и слово W. Для каждого файла fᵢ найти количество вхождений W.

### 4.1. Этап P — Разбиение

**Вид разбиения:** функциональное (function decomposition) — каждый файл обрабатывается независимо.

- Подзадача `i`: прочитать файл fᵢ, подсчитать вхождения W → вернуть count_i.
- Итого: **m подзадач** (по числу файлов).
- Полная независимость: подзадачи не делят данные, не взаимодействуют.

**Оценка P:** при m=100 и P=8 — 100 подзадач для 8 процессоров, достаточно. Размеры файлов могут различаться → нагрузка неравномерна → потребуется динамическое отображение.

### 4.2. Этап C — Коммуникация

- Список файлов и слово W рассылаются всем процессорам (broadcast, объём мал).
- Обмена промежуточными данными между подзадачами нет.
- После завершения каждого процессора: сбор count_i на главный узел (gather).

| Коммуникация | Тип | Объём |
|-------------|-----|-------|
| Рассылка имён файлов + W | broadcast | ~KB |
| Сбор count_i | gather | m × sizeof(long) |

**Оценка C:** объём коммуникации ничтожен по сравнению с I/O и вычислениями → задача коммуникационно дешёвая (communication-efficient).

### 4.3. Этап A — Агрегирование

Если m >> P, назначаем каждому процессору несколько файлов. Стратегия:

- **Статическое:** первые m/P файлов — процессору 0, следующие m/P — процессору 1 и т.д. Проблема: неравномерность размеров файлов.
- **Динамическое (рекомендуется):** пул задач (очередь файлов); процессор берёт следующий файл как только заканчивает предыдущий.

**Оценка A:** динамическое агрегирование предпочтительно; накладные расходы на выдачу задач минимальны (один файл — одна задача).

### 4.4. Этап M — Отображение

Динамическое: главный поток поддерживает очередь файлов; P рабочих потоков берут из очереди файл, обрабатывают, записывают результат, берут следующий.

```
Главный поток:  Queue<String> files = new ConcurrentLinkedQueue<>(fileList)
Рабочий поток:  while ((file = files.poll()) != null) { counts[file] = countWord(file, W); }
```

**Оценка M:** динамическое отображение обеспечивает балансировку при разных размерах файлов. Масштабируется при увеличении числа файлов.

### 4.5. Сравнение задач 3 и 4

| Критерий | Матрица × вектор (З3) | Подсчёт слов (З4) |
|----------|:---------------------:|:------------------:|
| Тип разбиения | По данным (строки) | Функциональное (файлы) |
| Нагрузка подзадач | Равномерная | Неравномерная |
| Коммуникация | Broadcast x (O(n)) | Минимальная |
| Балансировка | Статическая | Динамическая |
| Тип задачи | Compute-bound | I/O-bound |

## 5. Задание 5. Автостоянка — `synchronized`

**Условие:** несколько машиномест (SPOTS = 5). Автомобили — потоки. Если мест нет — ждать. Синхронизация через ключевое слово `synchronized`.

```java
// ParkingLotSync.java
public class ParkingLotSync {

    static final int SPOTS = 5;
    static final int CARS  = 10;

    static class ParkingLot {
        private final boolean[] spots = new boolean[SPOTS]; // true = занято
        private int free = SPOTS;

        synchronized int park(String car) throws InterruptedException {
            while (free == 0) {
                System.out.printf("  %-10s мест нет, ожидает...%n", car);
                wait();                        // освобождает монитор
            }
            for (int i = 0; i < SPOTS; i++) {
                if (!spots[i]) {
                    spots[i] = true;
                    free--;
                    System.out.printf("  %-10s занял место №%d  (свободно: %d)%n", car, i + 1, free);
                    return i;
                }
            }
            return -1;
        }

        synchronized void leave(String car, int spot) {
            spots[spot] = false;
            free++;
            System.out.printf("  %-10s покинул место №%d  (свободно: %d)%n", car, spot + 1, free);
            notifyAll();
        }
    }

    static class Car extends Thread {
        private final ParkingLot lot;
        private final int parkTimeMs;

        Car(String name, ParkingLot lot, int parkTimeMs) {
            super(name);
            this.lot = lot;
            this.parkTimeMs = parkTimeMs;
        }

        @Override
        public void run() {
            try {
                int spot = lot.park(getName());
                Thread.sleep(parkTimeMs);
                lot.leave(getName(), spot);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }

    public static void main(String[] args) throws InterruptedException {
        System.out.println("=== Задание 5. Автостоянка (synchronized) ===");
        System.out.printf("Мест: %d, автомобилей: %d%n%n", SPOTS, CARS);

        ParkingLot lot = new ParkingLot();
        Thread[] cars = new Thread[CARS];
        int[] times = {1200, 900, 1500, 800, 1100, 700, 1300, 600, 1000, 1400};

        for (int i = 0; i < CARS; i++)
            cars[i] = new Car("Авто-" + (i + 1), lot, times[i]);
        for (Thread c : cars) c.start();
        for (Thread c : cars) c.join();

        System.out.println("\nВсе автомобили покинули стоянку.");
    }
}
```

- `synchronized` на методах `park` и `leave` обеспечивает взаимное исключение.

- `notifyAll()` при освобождении места пробуждает всех ожидающих; один из них займёт место, остальные снова заблокируются в `while`.

## 6. Задание 6. Автостоянка — `Semaphore`

**Идея:** `Semaphore(SPOTS)` содержит SPOTS разрешений. `acquire()` — взять разрешение (или заблокироваться). `release()` — вернуть разрешение.

```java
// ParkingLotSemaphore.java
import java.util.concurrent.Semaphore;
import java.util.concurrent.atomic.AtomicInteger;

public class ParkingLotSemaphore {

    static final int SPOTS = 5;
    static final int CARS  = 10;

    static class ParkingLot {
        // fair=true: потоки обслуживаются в порядке FIFO (без «голодания»)
        private final Semaphore semaphore = new Semaphore(SPOTS, true);
        private final AtomicInteger spotCounter = new AtomicInteger(0);

        void park(String car, int parkTimeMs) throws InterruptedException {
            System.out.printf("  %-10s пытается занять место (доступно: %d)...%n",
                    car, semaphore.availablePermits());
            semaphore.acquire();
            int spot = spotCounter.incrementAndGet();
            System.out.printf("  %-10s занял место №%d  (доступно: %d)%n",
                    car, spot % SPOTS + 1, semaphore.availablePermits());

            Thread.sleep(parkTimeMs);

            System.out.printf("  %-10s покидает место №%d  (доступно: %d)%n",
                    car, spot % SPOTS + 1, semaphore.availablePermits() + 1);
            semaphore.release();
        }
    }

    // ... (Car, main — аналогично)
}
```

## 7. Задание 7. Автостоянка — условная переменная (`ReentrantLock` + `Condition`)

**Идея:** `ReentrantLock` заменяет монитор, `Condition spotAvailable = lock.newCondition()` — условную переменную. `await()` ≈ `wait()`, `signal()` ≈ `notify()`.

```java
// ParkingLotCondition.java
import java.util.concurrent.locks.*;

public class ParkingLotCondition {

    static final int SPOTS = 5;
    static final int CARS  = 10;

    static class ParkingLot {
        private int free = SPOTS;
        private final boolean[] spots = new boolean[SPOTS];

        private final Lock         lock           = new ReentrantLock();
        private final Condition    spotAvailable  = lock.newCondition();

        int park(String car) throws InterruptedException {
            lock.lock();
            try {
                while (free == 0) {
                    System.out.printf("  %-10s мест нет, ожидает...%n", car);
                    spotAvailable.await();
                }
                for (int i = 0; i < SPOTS; i++) {
                    if (!spots[i]) {
                        spots[i] = true;
                        free--;
                        System.out.printf("  %-10s занял место №%d  (свободно: %d)%n",
                                car, i + 1, free);
                        return i;
                    }
                }
            } finally {
                lock.unlock();
            }
            return -1;
        }

        void leave(String car, int spot) {
            lock.lock();
            try {
                spots[spot] = false;
                free++;
                System.out.printf("  %-10s покинул место №%d  (свободно: %d)%n",
                        car, spot + 1, free);
                spotAvailable.signalAll();
            } finally {
                lock.unlock();
            }
        }
    }

    // ... (Car, main — аналогично)
}
```

## 8. Задание 8. Автостоянка — `java.util.concurrent` (Semaphore + timeout)

**Условие:** если все места заняты, автомобиль ждёт не более MAX_WAIT мс, затем уезжает на другую стоянку. **Запрещено** использовать слово `synchronized`.

**Реализация:** `Semaphore.tryAcquire(timeout, TimeUnit)` — возвращает `false`, если за отведённое время разрешение не получено.

```java
// ParkingLotConcurrent.java
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class ParkingLotConcurrent {

    static final int  SPOTS       = 5;
    static final int  CARS        = 10;
    static final long MAX_WAIT_MS = 1500L;

    static class ParkingLot {
        private final Semaphore       semaphore    = new Semaphore(SPOTS, true);
        private final AtomicInteger   spotAssigner = new AtomicInteger(0);

        void park(String car, int parkTimeMs) throws InterruptedException {
            System.out.printf("  %-10s ожидает место (макс. %d мс)...%n", car, MAX_WAIT_MS);

            boolean acquired = semaphore.tryAcquire(MAX_WAIT_MS, TimeUnit.MILLISECONDS);
            if (!acquired) {
                System.out.printf("  %-10s [УЕХАЛ]  — ждал > %d мс, мест не дождался%n",
                        car, MAX_WAIT_MS);
                return;
            }

            int spot = spotAssigner.incrementAndGet() % SPOTS + 1;
            System.out.printf("  %-10s занял место №%d  (доступно: %d)%n",
                    car, spot, semaphore.availablePermits());

            Thread.sleep(parkTimeMs);

            System.out.printf("  %-10s покидает место №%d  (доступно: %d)%n",
                    car, spot, semaphore.availablePermits() + 1);
            semaphore.release();
        }
    }

    // ... (Car, main — аналогично)
}
```

## 9. Сравнительный анализ механизмов синхронизации

| Механизм | Задание | Простота | Гибкость | Timeout | Fair | Рекомендуется |
|----------|:-------:|:--------:|:--------:|:-------:|:----:|:-------------:|
| `synchronized` + `wait/notifyAll` | 5 | Средняя | Низкая | Нет | Нет | Простые случаи |
| `Semaphore(n, fair)` | 6 | Высокая | Средняя | Только tryAcquire | Да | Пул ресурсов |
| `ReentrantLock` + `Condition` | 7 | Средняя | Высокая | `tryLock` | Да | Сложные условия |
| `Semaphore.tryAcquire(t, u)` | 8 | Высокая | Высокая | **Да** | Да | Таймаут + без sync |

**Общие рекомендации:**
- Если нужно ограничить доступ к пулу ресурсов фиксированного размера — **`Semaphore`**.
- Если нужно несколько независимых условий на один замок — **`ReentrantLock` + несколько `Condition`**.
- Если автомобиль (поток) не должен ждать дольше заданного времени — **`tryAcquire(timeout)`**.
- `synchronized` остаётся актуальным для простых случаев взаимного исключения одного ресурса.
